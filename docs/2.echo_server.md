# 02 - Building an Echo Server

## Goal

Before implementing Redis, I need to understand the most fundamental component of any database server:

```text
Client
  ↓
TCP Connection
  ↓
Server
  ↓
Response
```

Redis is ultimately a TCP server that receives bytes, processes them, and sends bytes back.

The objective of this milestone was to build the simplest possible TCP server:

```text
Client sends message
        ↓
Server receives message
        ↓
Server sends same message back
```

This pattern is known as an Echo Server.

---

## First Principles

A server is simply a process that:

1. Opens a port
2. Waits for clients to connect
3. Receives data
4. Processes data
5. Sends a response

Nothing more.

Conceptually:

```text
while true:
    wait for connection

    while connection is alive:
        read bytes
        process bytes
        write bytes
```

The network does not understand commands.

The network only understands bytes.

Everything higher level is built on top of this idea.

---

## Server Startup

The server starts by listening on a configured host and port.

```go
net.Listen("tcp", host+":"+port)
```

This creates a listening socket.

Conceptually:

```text
Operating System
        ↓
Port 7379
        ↓
Waiting for connections
```

No communication happens until a client connects.

---

## Accept Loop

The server continuously waits for incoming connections.

```go
for {
    conn, err := listener.Accept()
}
```

Important observation:

```text
Accept() is blocking.
```

This means:

```text
No client connected
        ↓
Server sleeps
        ↓
Client connects
        ↓
Accept returns
```

The operating system wakes the server only when a new connection arrives.

---

## Connection Lifecycle

Once a client connects:

```text
Client
   ↓
TCP Connection Established
   ↓
Server Receives net.Conn
```

The connection object represents a bidirectional communication channel.

Using this object:

```go
conn.Read(...)
conn.Write(...)
```

the server can exchange bytes with the client.

---

## Reading Data

The server allocates a buffer:

```go
buf := make([]byte, 512)
```

and reads data from the connection:

```go
n, err := conn.Read(buf)
```

Important lesson:

```text
TCP is a byte stream.
```

TCP does NOT preserve messages.

Many beginners incorrectly assume:

```text
One Write()
        ↓
One Read()
```

This is not guaranteed.

A command may arrive:

```text
In one read
```

or

```text
Split across multiple reads
```

or

```text
Multiple commands may arrive together
```

For the echo server, a single 512-byte read is sufficient.

For Redis, a proper protocol parser will eventually be required.

---

## Writing Data

The response logic is straightforward:

```go
conn.Write([]byte(cmd))
```

Whatever the client sends is immediately returned.

Example:

```text
Client -> hello
Server -> hello
```

This verifies that:

* TCP connection works
* Reads work
* Writes work

---

## Current Architecture

```text
Client
   ↓
TCP Socket
   ↓
Accept()
   ↓
Read()
   ↓
Echo Response
   ↓
Write()
```

At this stage there is no:

* Command parsing
* Database
* Key-value store
* Redis protocol

Only networking.

---

## Limitation of Current Design

The server is synchronous.

Current flow:

```text
Accept Client A
        ↓
Handle Client A forever
        ↓
Accept Client B
```

This means:

```text
Client A connected
        ↓
Client B cannot be accepted
```

until Client A disconnects.

Only one client can be actively served at a time.

This is acceptable for learning networking fundamentals but is not how Redis operates.

Future iterations will introduce:

```text
Multiple clients
```

and eventually:

```text
Event loops
I/O multiplexing
epoll
```

---

## What I Learned

### 1. A database starts as a TCP server

Before key-value stores, persistence, or replication, Redis is simply a server listening on a TCP port.

---

### 2. Networking is fundamentally bytes

Applications think in terms of commands.

Networks think in terms of bytes.

A protocol is required to translate between the two.

---

### 3. TCP is a stream, not a message queue

A single command may be fragmented across multiple reads.

Future Redis commands cannot rely on one Read() call.

---

### 4. Blocking calls are everywhere

Both:

```go
Accept()
Read()
```

block until work becomes available.

The operating system handles the waiting.

---

### 5. Concurrency problem discovered

The current server can only actively serve one client at a time.

This limitation motivates the next phase:

```text
Concurrent Connections
        ↓
Event Loops
        ↓
Redis Architecture
```
