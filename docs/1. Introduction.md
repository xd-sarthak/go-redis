# Redis Clone in Go - Introduction

## Why Build Redis?

I want to understand how high-performance databases and distributed systems work internally instead of treating them as black boxes.

Redis looks simple from the outside:

```text
SET name sarthak
GET name
```

but internally it involves:

* Networking
* TCP sockets
* Event loops
* Protocol design
* Concurrency
* Memory management
* Persistence
* Replication
* Distributed systems concepts

The goal of this project is not to create a production-ready Redis replacement. The goal is to understand the engineering principles behind one of the most successful infrastructure systems ever built.

---

## First Principles

A database server is fundamentally just:

```text
Client
  ↓
Network
  ↓
Server
  ↓
Memory / Storage
  ↓
Response
```

Everything else is an optimization.

Redis receives bytes from clients, interprets those bytes as commands, performs operations on data, and sends bytes back as responses.

At its core:

```text
Input Bytes
    ↓
Parse
    ↓
Execute
    ↓
Output Bytes
```

---

## Redis Architecture Overview

One of the most surprising things about Redis is that it is primarily single-threaded for command execution.

Many people assume:

```text
More Threads = More Performance
```

Redis demonstrates that this is not always true.

Instead of using many threads to execute commands simultaneously, Redis uses:

```text
Many Connections
        ↓
     Event Loop
        ↓
Single Execution Thread
```

This design avoids:

* Locks
* Deadlocks
* Race conditions
* Lock contention

while still handling thousands of clients concurrently.

---

## Why Single Threaded?

Imagine multiple threads modifying the same data structure:

```go
map[string]string
```

If two threads execute:

```text
INCR counter
```

at the same time, synchronization becomes necessary.

The system now requires:

* Mutexes
* Atomic operations
* Lock management

Complexity grows rapidly.

Redis avoids this entirely.

A single thread owns the database state.

Only that thread can modify:

* Keys
* Values
* Expiration metadata
* Transactions
* Pub/Sub structures

Because only one thread touches the data:

```text
No races
No locks
No deadlocks
```

---

## How Does Redis Handle Concurrency?

A common misconception is:

```text
Single Thread = One Client At A Time
```

This is false.

Redis can maintain tens of thousands of active TCP connections simultaneously.

The trick is that most clients spend their time waiting.

Example:

```text
100,000 connected clients

99,990 idle
10 sending commands
```

The server does not need 100,000 threads.

Instead, it uses I/O multiplexing.

---

## I/O Multiplexing

Redis relies on operating system facilities such as:

* epoll (Linux)
* kqueue (BSD/macOS)
* select/poll (older systems)

Instead of asking:

```text
Does Socket A have data?
Does Socket B have data?
Does Socket C have data?
```

the server asks the kernel:

```text
Tell me when any socket becomes ready.
```

The kernel monitors all connections and wakes Redis only when work exists.

Conceptually:

```text
Many TCP Connections
        ↓
       Kernel
        ↓
      epoll
        ↓
    Event Loop
```

---

## Request Lifecycle

Suppose a client sends:

```text
GET user:123
```

The sequence is:

```text
Client sends bytes
        ↓
Kernel receives bytes
        ↓
epoll reports socket ready
        ↓
Redis event loop wakes up
        ↓
Redis reads request
        ↓
Redis parses protocol
        ↓
Redis executes command
        ↓
Redis writes response
        ↓
Redis waits again
```

Notice that Redis itself is not waiting on sockets.

The operating system handles the waiting.

Redis spends most of its time doing useful work.

---

## Why Is This Fast?

Most Redis commands are extremely small.

Example:

```text
GET key
SET key value
INCR counter
```

Execution often takes microseconds.

Because command execution is so fast, a single CPU core can process a massive number of requests per second.

The bottleneck is often network latency rather than computation.

---

## Important Tradeoff

Since commands execute on a single thread:

```text
GET A
GET B
SET C
INCR D
```

are executed sequentially.

This gives Redis an important property:

```text
Commands are atomic.
```

However, it also introduces a limitation.

A slow command can block the entire server.

Example:

```text
KEYS *
```

on millions of keys.

While Redis executes that command:

```text
GET user
SET name
INCR counter
```

from other clients must wait.

Redis trades parallel execution for simplicity and predictable performance.

## Current Mental Model

```text
Many TCP Connections
        ↓
Kernel (epoll)
        ↓
Event Loop
        ↓
Single Command Execution Thread
        ↓
In-Memory Database
        ↓
Response
```

Redis does not achieve concurrency by executing many commands simultaneously.

Redis achieves concurrency by efficiently managing many waiting clients while executing commands one at a time.
