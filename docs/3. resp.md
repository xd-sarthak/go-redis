# 03 - RESP (Redis Serialization Protocol)

## Why Does Redis Need RESP?

At first glance, Redis commands look simple:

```text
SET name sarthak
GET name
PING
```

However, computers do not send commands across the network.

They send bytes.

A Redis server receiving bytes from a TCP connection must answer:

```text
Where does a command start?
Where does it end?
How many arguments are there?
How long is each argument?
```

This is the problem solved by RESP (Redis Serialization Protocol).

RESP defines a standard format for encoding data between clients and servers.

Conceptually:

```text
Application Command
        ↓
RESP Encoding
        ↓
TCP Bytes
        ↓
RESP Decoding
        ↓
Application Command
```

Without RESP, Redis would have no reliable way to interpret incoming bytes.

---

## First Principles

TCP is a stream protocol.

This means:

```text
Client:
SET name sarthak

Network:
SE
T na
me sa
rtha
k
```

or

```text
SET name sarthakGET age
```

Both are valid.

TCP guarantees ordered delivery of bytes.

TCP does NOT guarantee message boundaries.

Therefore Redis needs a protocol that explicitly describes:

```text
Data Type
Length
Payload
```

RESP provides exactly this.

---

## RESP Data Types

RESP uses a single prefix character to indicate the type of the following data.

### Simple String

Prefix:

```text
+
```

Example:

```text
+OK\r\n
```

Represents:

```text
OK
```

---

### Error

Prefix:

```text
-
```

Example:

```text
-ERR unknown command\r\n
```

Represents:

```text
ERR unknown command
```

---

### Integer

Prefix:

```text
:
```

Example:

```text
:100\r\n
```

Represents:

```text
100
```

---

### Bulk String

Prefix:

```text
$
```

Example:

```text
$5\r\nhello\r\n
```

Structure:

```text
$
Length
\r\n
Payload
\r\n
```

Represents:

```text
hello
```

---

### Array

Prefix:

```text
*
```

Example:

```text
*2\r\n
$4\r\n
PING\r\n
$4\r\n
PONG\r\n
```

Represents:

```go
[]interface{}{
    "PING",
    "PONG",
}
```

Arrays are the most important RESP type because Redis commands are transmitted as arrays.

Example:

```text
SET name sarthak
```

becomes:

```text
*3\r\n
$3\r\n
SET\r\n
$4\r\n
name\r\n
$7\r\n
sarthak\r\n
```

---

## My Parser Architecture

The parser follows a recursive descent approach.

Everything begins at:

```go
Decode()
```

which delegates to:

```go
DecodeOne()
```

Architecture:

```text
Decode
  │
  ▼
DecodeOne
  │
  ├── '+' → readSimpleString
  │
  ├── '-' → readError
  │
  ├── ':' → readInt64
  │
  ├── '$' → readBulkString
  │
  └── '*' → readArray
```

The first byte determines which parser is invoked.

---

## The Core Idea: Delta

Every parser function returns:

```go
(value, delta, error)
```

Example:

```go
("hello", 11, nil)
```

Meaning:

```text
Parsed Value = "hello"

Consumed 11 bytes

No Error
```

The parser never modifies the original buffer.

Instead, it advances through the buffer by tracking how many bytes have been consumed.

This allows recursive parsing of nested structures.

---

## Understanding readLength()

Many RESP structures contain a length field.

Examples:

```text
$5\r\nhello\r\n
```

and

```text
*3\r\n
...
```

The parser uses:

```go
readLength()
```

to extract these numbers.

Example:

Input:

```text
5\r\nhello\r\n
```

Output:

```go
length = 5
delta = 3
```

Why delta = 3?

Because:

```text
5\r\n
```

consumes three bytes.

This allows the parser to jump directly to:

```text
hello
```

without reparsing the length field.

---

## Parsing a Bulk String

Input:

```text
$5\r\nhello\r\n
```

Flow:

```text
readBulkString
        ↓
readLength
        ↓
length = 5
        ↓
extract 5 bytes
        ↓
"hello"
```

Result:

```go
"hello"
```

---

## Parsing an Array

Input:

```text
*2\r\n
$4\r\n
PING\r\n
$4\r\n
PONG\r\n
```

Flow:

```text
readArray
        ↓
readLength
        ↓
count = 2
        ↓
DecodeOne()
        ↓
readBulkString()
        ↓
"PING"
        ↓
DecodeOne()
        ↓
readBulkString()
        ↓
"PONG"
```

Result:

```go
[]interface{}{
    "PING",
    "PONG",
}
```

---

## Why Arrays Use DecodeOne()

The most elegant part of the parser is:

```go
elem, delta, err := DecodeOne(data[pos:])
```

inside:

```go
readArray()
```

Instead of assuming arrays contain strings, the parser delegates parsing back to DecodeOne.

This means arrays can contain:

```text
Strings
Integers
Errors
Bulk Strings
Arrays
```

including nested arrays.

Example:

```text
*2
:42
+hello
```

or

```text
*2
*2
:1
:2
+done
```

work automatically.

This recursive structure is what makes RESP easy to parse.

---

## Parsing a Redis Command

Input:

```text
SET name sarthak
```

Actually arrives as:

```text
*3\r\n
$3\r\n
SET\r\n
$4\r\n
name\r\n
$7\r\n
sarthak\r\n
```

Parser result:

```go
[]interface{}{
    "SET",
    "name",
    "sarthak",
}
```

At this point the networking layer no longer deals with bytes.

The command executor receives a structured representation of the command.

Conceptually:

```text
TCP Bytes
      ↓
RESP Decoder
      ↓
Command Array
      ↓
Command Executor
```

## Limitations of Current Implementation

The current parser is sufficient for basic Redis commands but is not production-ready.

Known issues:

### Missing Bounds Checks

Malformed input can cause:

```text
index out of range
```

panics.

---

### No Negative Integer Support

RESP integers such as:

```text
:-42\r\n
```

are not handled.

---

### No Null Bulk Strings

RESP supports:

```text
$-1\r\n
```

which represents:

```text
null
```

Current implementation does not support this.

---

### Unknown Types Are Silently Accepted

Current behavior:

```go
return nil, 0, nil
```

for unknown prefixes.

This should instead return an error.

## What I Learned

### 1. Protocols Exist Because TCP Is A Stream

TCP only transports bytes.

Protocols provide structure and meaning.

---

### 2. RESP Is Extremely Simple

Every value begins with a type marker.

The rest of the parsing logic follows naturally.

---

### 3. Parsing Is Mostly Cursor Management

The parser's primary job is:

```text
Read Value
Advance Cursor
Repeat
```

The delta returned by each parser function makes this possible.

---

### 4. Recursion Makes Arrays Elegant

By allowing arrays to call DecodeOne again, the parser automatically supports nested RESP values.

---

### 5. Redis Commands Are Just Arrays

The command:

```text
SET name sarthak
```

is not special.

It is simply a RESP array containing three bulk strings.

This realization makes the separation between:

```text
Protocol Layer
```

and

```text
Command Execution Layer
```

much clearer.

---
